Getting Started with Servo

Welcome to the world of chatbots. In this tutorial, we’ll learn how to create a simple Alexa bot in the Servo system. The bot we’ll build here calculates a square root of any given number. Let's call it Square With Me. We will learn through this example how to use - and think in - behavior trees, how to deal with Alexa imperfections and limitations and above all, how does it feel when you create a new intelligent creature.

Well, the last goal will require some imagination. But let’s get started.

Setup

The Servo system needs as prerequisites NodeJs and Git. Install Node and Git using their public installers, depending on the operating system you use.

To get Servo code, clone the repository from bitbucket:

git clone https://<username        >@bitbucket.org/servoai/servo-admin.git

cd servo-admin

Servo comprises of two distinct parts that works together: the editor, in which you create and maintain the behavior trees, and the run-time, which is responsible to (a) maintaining the trees for the editor (save, load, publish) and (b) the bot engine, which runs the behavior trees and connects to the various chat clients, such as Facebook and Alexa.

Hello Behavior Trees

So, now it's time to create our first behavior tree. In the editor, go to Projects. Then, click New Project:

Fig 1

Let's call this project square-with-me. We should avoid spaces [b]  in all names and types, in order to avoid weird file system issues, as the names become files and folders. We use snake case out of convenience, but this is up to you. For custom node types Servo uses Polish Case.

Every project has root tree - it's called FSM in Servo lingo, standing for Full Scale Machine (I guess the acronym meaning was more an afterthought; we started with Finite State Machine; but on that, some other time). Under it, we are going to put logic composite nodes and actions.

Now, let's stop for a second and think what we want from our bot. Here’s the very short list:

  • It should greet the user and ask it for a number
  • It should validate that the response is a number, and if not, explain again what is needed
  • Once the number is received, it should calculate its square root and tell it back to the user

I haven't yet included other peripheral tasks, like the ability to provide more detailed instructions and information, the ability to cancel the operation at any point, the ability to chit-chat, and more. We can deal with all that later.

How we do all that using behavior trees? As you’ll see, it’s pretty simple. Once you adapt your brain to think this way, you won't want to go back.

So behavior trees has four node types: Composites, Conditions, Actions and Decorators.

Composites are the nodes in the middle of the tree. They control the flow, as we will see in a minute. Conditions and Actions are the leafs. Composites can have several children while Conditions and Actions have none. Decorators are like filters - they have one child and modify whatever passes in them.

All nodes can only have one parent, which makes it really easy to draw a tree.

Here’s an example of how it looks. We’ll call it Figure 2:

Each node is really a class, an object, which has an execution method that performs various tasks and returns Success or Failure. The behavior tree starts executing each time we have an input from the user. The root is executed with the input text as a parameter, then the execution propagates down: each node is executing its children from left to right. While the execution isn't finished yet - for example, if a database call still hasn't returned a value - the node returns Running.

So in our tree we want to ask for a number, then examine the variable type received. If it is not a number, output an error message. Otherwise, output the square root of the number. We really want to make a small program like this:

Output “what is your number?”

Wait until there is user input

x = user input

if (typeof x ==number)

     output square root of (x)

else

     output “not a number!”

Let’s delve on the tree from Fig 2.

We use here two composites - called Sequence( the → ) and a PrioritySelector (the ? ). A Sequence executes all its children in sequence, from left to right. If one of them fails (ie returns a Failure), the sequence node fails too. If all of them succeeds, the sequence returns Success. In that, it's like an AND operator. A PrioritySelector is like an OR: it tries to execute its children from left to right and stops when there's a Success. It then returns a Success  as well. If none succeeds, it returns a Failure.

We also need a Condition node - we’ll call it  EntityCondition - and  two Action types - one to ask a question and wait for an answer, called AskAndWait, and one named GeneralMessage. The InputType compares a variable (eg. x ) to a certain type string (eg. “number”) and returns Success if they are equal, Failure otherwise. The AskAndWait returns Running until the user has answered the question asked, and then it returns Success. The GeneralMessage outputs a message and returns a Success.

How does it work? As mentioned, our behavior trees are executed much like English: from left to right, top to bottom. If you follow the execution from the root, it first executes a Sequence, which has two children: an Action and a PrioritySelector. The Action is executed first. It is an AskAndWait-type action, which asks a question and waits there, on that node, until an answer from the user comes in. At that point the execution continues to the next child of the Sequence, which is the PrioritySelector.

PrioritySelectors are really an IF-THEN-ELSE statements: they try their children from left to right until one of them succeeds. So this PrioritySelector goes first to the left child. That child is a Sequence which runs IF (is a number) THEN “output square root”. It does it by first going into the Condition “is a number”. This condition - conditions are shown elliptical - examines the message received from the user. If its type is a number (1, 23, one hundred and two etc.) it returns Success, so the Sequence continues into the next child, which is an Action, which outputs the square root, and returns a Success [c] . Then, the Sequence itself will return a Success to its parent, a PrioritySelector.

Now we have a child to to a PrioritySelector that returns a success, so the execution need not continue, and it returns a Success too. Once it does, the upper Sequence finished all its children, and we are done with our tree.

The bot pipeline

Bots these days really exploit a pipeline of technologies to do their thing. If we are using a speech platform such as Alexa, then it would look like this:

Once a user utters a sentence, it goes into the ASR - Automatic Speech Recognition which outputs a sentence in text. This sentence then serve as an input to the NLU - the Natural Language Understanding module, which outputs an intent  and slots . These are the ones fed into Servo. Servo uses its behavior tree logic to output text into the Text To Speech engine, if exists.

There are plenty of ASR, NLU and TTS libraries, ranging from open source (eg rasa.ai), online (Amazon, google, Microsoft, IBM Watson) to paid products (eg Nuance). Sometimes they are combined together - for example, on Alexa:

Intent and slots are identifiers that tells us what the user meant in her sentences. For example, an intent ‘hello’ would be the output for ‘hi’, ‘hello’, ‘yo’, ‘hey’, ‘greetings’ and the like. An intent ‘checkFlightStatus’ would be calculated for ‘when’s my flight’, ‘what time does my flight leave’, ‘please give me the time of departure for American Airlines number 100’, etc.

How does the NLU know to produce these intents? The NLU has a model, which the developer should train, which is a sample list of sentences and intent ids, all defined by you, the developer, at the specific engine you use - Amazon, in this instance. Slots (sometimes also called Entities) are like return values for certain recognized values. For example, on ‘please give me the time of departure for American Airlines number 100’ there would be an ‘airline’ slot with value ‘American Airline’ and ‘flightNumber’ with 100. The naming is like variable names - it’s your choice.

This is done by providing the NLU library with a list of example sentences. There usually is a GUI to do that.

Let’s build here, as an exercise, a very simple NLU model for our very simple bot.

Let’s say we won’t to support sentences more or less like these:

“Please calculate square root of 100”

“Square root 220”

“What’s the root of 300”

“Do square root of 11”

“Figure out the root of 10”

“Can you compute square root of 4 million”

All of these, to a human, have one intent: “calculate the square root of the number given”

We need to NLU engines we provide (either at some UI or by a file) these samples, and invent a unique id for them - the intent ID. Let’s call it “ SquareMe ”. So the file would look something like:

SquareMe: Please calculate square root of 100”

SquareMe: Square root 220”

SquareMe: What’s the root of 300”

SquareMe: Do square root of 11”

SquareMe: Figure out the root of 10”

SquareMe: Can you do the calculation of square root of 4 million”

If you you trained with these sentences a good NLU engine, it should recognize an utterance (=sentence) saying something like

“Would you be able to compute the root of 99?”

It would give you an intent ID of SquareMe

What about the number? Same idea: we tell the engine the entity we need to identify

SquareMe: Please calculate square root of {number}

SquareMe: Square root { number }

SquareMe: What’s the root of { number }

SquareMe: Do square root of { number }

SquareMe: Figure out the root of { number }

SquareMe: Can you do the calculation of square root of {number}

NLU engines already have the needed general built-in entities, like numbers, cities, currencies etc, so they would know to reg o nize 99 from the above example, and output it as an entity whose name is number .

That’s a simplified version of NLU training. Once you do it over the Alexa setup (see below), you’ll get the jist of it.

In some channels, for example, on facebook, we get the raw text. On Alexa we only receive the data post-ASR and post-NLU. Now we can send a response. For example, lets say the user said “I agree” (which is not a good response, because we were asked for a number). If that is the case, the NLU would output intent = yes. Now To simulate that, add a field intentId:”yes” to the To  object and click Send

An interesting thing will happen: you will see the upper From box changes to the “bad input” message, and then “please provide a number, any number”

[e]

What happened here?

First, the wakeup call indeed waked up the tree, and it started ticking. It went into AskAndWait and did the asking part - asking for the number - that went into the From . The tick stopped there, until we sent the response from the To . Then, it proceeded, the condition ‘is it a number’ failed, and the execution continued to the ‘bad output’ - which indeed, arrived to the From . The tree then finished - and started again, half a second later! And when it did, it went into the AskAndWait and sent the initial question again.

Let’s now provide a good answer. For that, we should add an entity array to the data:

Some internals

If you want to understand a bit more - always a good approach, if you have the time - he AskAndWait works in the following way:

Its execution method - called ‘tick’ is executed every pre-configured interval, defaulted to 500 ms. It then returns Running for all the the executions, all the ticks. Once a user message arrives, the AskAndWait finally returns Success, so its parent Sequence can continue.
